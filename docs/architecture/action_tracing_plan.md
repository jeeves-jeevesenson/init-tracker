# Action Tracing & Idempotency Plan

## Goal
To ensure that player actions are reliably tracked, deduplicated, and visible in latency forensics, especially when the main thread is blocked by heavy broadcast work.

## Tracing Architecture
We use the existing live debug JSONL trace infrastructure. Actions are traced through the following lifecycle:

1. **ws.message.received**: Recorded immediately when the FastAPI websocket thread receives a message.
   - Includes `action_id` and `trace_id`.
   - Records `received_at_ns`.
2. **ws.action.queued**: Recorded when the action is placed into the `self._actions` queue for the Tk thread.
3. **ws.action.dispatch.start**: Recorded when the Tk thread pops the action from the queue.
   - **New field**: `queue_wait_ms` = `dispatch_start - received_at`.
   - **New field**: `queue_size` (current backpressure).
   - **New field**: `slow_queue_wait` (boolean, true if > 5000ms).
4. **ws.action.dispatch.end**: Recorded when the action handler (e.g., `_lan_apply_action`) completes.

## Idempotency Contract
To prevent duplicate execution during long lag windows:

- **Scope**: Idempotency is scoped by `action_id` (provided by client or generated by server).
- **History**: `LanController` maintains an `_action_states` dictionary (limit: 500 entries).
- **States**:
  - **pending**: Action is in the queue or currently being dispatched. Duplicate receipt returns an "already queued" ack.
  - **completed**: Action has finished. Duplicate receipt returns the cached result status.

## Immediate Acknowledgement
- Upon receipt of a valid action message, the server IMMEDIATELY sends an `action_ack` with `status: "accepted"`.
- This allows the frontend to show "Processing..." state even if the Tk thread is blocked and hasn't started dispatch yet.
- A final `action_ack` with `status: "completed"` is sent once dispatch finishes.

## Frontend Requirements
- LAN UI must generate a stable `action_id` for each user attempt.
- LAN UI should show a visual indicator (e.g., "Processing...") while an action is in flight.
- Retries (if any) should reuse the same `action_id`.
