# BUG-20260604-SMOKE-01

## Title
Browser Smoke Harness crash on AoE actions (KeyError: 'active')

## Scenario
`black-tan-combat-exploration`

## Artifact Path
`logs/browser-smoke/black-tan-combat-exploration/20260604_160539`

## Exact Failure Symptom
The harness crashes with `[ERROR] Exploration loop failed: 'active'` when a monster selects an AoE action (e.g., Vda-Scorcher's Flamethrower Burst).

## Reproduction Command
`env INIT_TRACKER_DEBUGGING=1 .venv/bin/python scripts/validation/browser-smoke-harness.py --scenario black-tan-combat-exploration --max-rounds 1 --max-turns 40 --artifact-root logs/browser-smoke --base-url http://127.0.0.1:8787`

## Observed Behavior
At Turn 6, the Vda-Scorcher selects "Flamethrower Burst". The harness executes `aoe = page.evaluate("window.__dmcontrolSmoke.aoeState()")` and then attempts to access `aoe["active"]`. Since the `aoeState()` hook in `assets/web/dmcontrol/index.html` returns the `aoePlacementMode` object directly (which lacks an `active` property), Python raises a `KeyError`.

## Expected Behavior
The harness should check for the existence of the `aoe` object itself, or the hook should be updated to return a property indicating activity (e.g., `{ active: true, ... }`).

## Relevant Event Log Excerpts
(Event log not written due to crash)

## Suspected Area
`scripts/validation/browser-smoke-harness.py` (line 249-250) and `assets/web/dmcontrol/index.html` (line 3888).

## Recommended Next Bounded Task
Update `scripts/validation/browser-smoke-harness.py` to check `if aoe:` instead of `if aoe and aoe["active"]:`, or normalize the `aoeState` hook.

## Fix Summary
- Normalized `aoeState` and `targetPreviewMode` hooks in `assets/web/dmcontrol/index.html` to always return an object with an `active` property (e.g., `{ active: true, ... }` or `{ active: false }`).
- Updated `scripts/validation/browser-smoke-harness.py` to use `.get("active")` defensively when checking AoE and Target Preview states.
- Verified that the harness no longer crashes with `KeyError: 'active'`.
